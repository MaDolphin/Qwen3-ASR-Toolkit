# 技术改进点文档

本文档用于说明当前项目相对于原始工具形态做了哪些技术升级、还存在什么问题，以及下一步建议的改进方向。

## 1. 已完成的技术改进

### 1.1 从单一 CLI 工具升级为服务化架构

原始形态更偏向本地脚本调用。

当前项目已经拆成两层：

- 核心工具层：`qwen3_asr_toolkit/`
- 服务层：REST + WebSocket

带来的收益：

- 更适合云端部署
- 更容易被前端、业务系统、第三方服务接入
- 便于统一鉴权、日志和监控

### 1.2 长音频 VAD 切分能力增强

当前离线转写会：

- 对长音频自动执行 VAD
- 强制单段不超过 `60s`
- VAD 异常时自动回退到固定长度切分

这解决了两个问题：

- 避免超长音频直接触发 vLLM 限制
- 降低切分不稳定导致的失败率

### 1.3 新增实时转写链路

项目现在同时支持：

- 离线转写
- 实时转写

实时模式通过：

- WebSocket 服务端
- 文件模拟流式 CLI

实现准实时增量转写，适合后续接浏览器录音、桌面采集、电话流转写等场景。

并且这次进一步升级为：

- WebSocket 新协议 `ready/start/partial/segment_final/final`
- 接收与推理解耦的队列式处理
- 当前语音段增量识别
- 静音后自动提交当前语音段

### 1.4 新增离线 Forced Aligner 支持

离线接口新增了显式参数：

- `use_forced_aligner`

只有传该参数才会执行对齐，符合产品边界。

并且已经适配远端 `Qwen3-ForcedAligner-0.6B` 的 vLLM 服务接口：

- `/tokenize`
- `/pooling`

### 1.5 配置统一收敛到 `.env`

原先配置散落在命令行参数和示例代码里。

当前已经统一为：

- 项目根目录 `.env`
- 自动加载

收益：

- 更适合服务器部署
- 更适合 CI/CD
- 更容易做多环境切换

### 1.6 上游 `qwen_asr` 包已迁入主项目

之前上游依赖存放在 `references/`。

现在已经：

- 将 `qwen_asr/` 迁入主仓库
- 迁出上游 example 快照
- 删除 `references/`

收益：

- 项目自包含
- 结构更清晰
- 后续维护不再依赖临时参考目录

### 1.7 测试补齐

当前已经补了基础测试，覆盖：

- 长音频分段约束
- VAD 回退逻辑
- 实时流式状态机
- REST/WS 协议
- `qwen_asr` 嵌入后的导入行为

## 2. 当前存在的问题

### 2.1 项目结构仍然偏重

虽然已经比之前清晰很多，但现在仓库同时存在：

- 主工具实现 `qwen3_asr_toolkit/`
- 上游迁入包 `qwen_asr/`
- 上游示例快照 `examples/upstream-qwen3-asr/`

这会让第一次接手的人不容易迅速理解“哪个是当前主路径”。

建议继续收敛：

- 把 README 和文档里明确标出“主路径”
- 对 `examples/upstream-qwen3-asr/` 增加更强的“参考用途”标识

### 2.2 Forced Aligner 后处理仍有优化空间

当前已经能返回真实时间戳，但仍存在现象：

- 部分 token 尾部时间可能相同
- 某些短词时间戳分布不够平滑

原因在于：

- 远程服务返回的是 `token_classify` 结果
- 当前客户端需要做 timestamp token 提取和修复

后续可以优化：

- 更细的单词级平滑
- 尾部重复时间压缩
- 基于片段时长做更强约束

### 2.3 文档和实现曾经不同步

项目历史里有过这种情况：

- 文档还写着 aligner 未部署
- 实现已经接上了 aligner

当前这次已经做了同步，但后续仍建议：

- 每次接口变化时同步更新 `README` 和 `doc/`
- 把验证结果写进文档

### 2.4 当前实时仍属于“准实时”，不是真正的官方 realtime

由于当前远端 vLLM 服务未暴露官方 `/v1/realtime` 端点，项目的实时模式仍然属于：

- 项目内自管协议
- 准实时处理
- 非模型原生 streaming state

这已经能明显改善当前体验，但和真正的官方 realtime 能力相比仍有差距。

### 2.5 生产部署能力刚补齐模板，但还没形成完整体系

当前仓库已经补了：

- `deploy/systemd/qwen3-asr-toolkit.service`
- `scripts/deploy_server.sh`

它们已经够支撑：

- 单机部署
- 开发环境
- 小规模服务

但对更完整的生产环境来说还可以继续增强：

- 日志切割
- 健康检查自动告警
- 灰度发布
- 多实例部署

## 3. 后续建议的技术改进

### 3.1 继续整理结构

建议把仓库角色分层得更明显：

- `qwen3_asr_toolkit/`：业务主实现
- `qwen_asr/`：嵌入上游包
- `examples/`：仅示例
- `doc/`：正式文档
- `scripts/`：运维脚本

### 3.2 增加系统级部署能力

建议继续补齐：

- `systemd` 单元文件
- Dockerfile
- `docker-compose.yml`

### 3.3 增加观测能力

建议后续增加：

- 请求耗时统计
- 每段切分数量统计
- 对齐耗时统计
- 错误分类日志

这样上线后更容易定位：

- 是 ASR 慢
- 还是 VAD 慢
- 还是 aligner 慢

### 3.4 继续增强实时模式

建议后续继续优化：

- 更稳健的静音检测
- 更细的段内文本冻结策略
- 若远端服务支持 `/v1/realtime`，切换到官方 realtime 模式

### 3.5 增加更真实的集成测试

当前单元测试已经够用，但还可以再加：

- 基于 `sample/` 的集成测试脚本
- REST + WS 真实链路测试
- 长音频带对齐测试基准

## 4. 当前结论

这个项目已经从“本地调用脚本”升级成了“可部署、可调用、可测试、可维护”的服务化工具。

接下来最值得继续投入的方向不是再堆功能，而是：

- 结构继续收敛
- 文档持续同步
- 对齐后处理继续优化
- 生产部署能力补齐
